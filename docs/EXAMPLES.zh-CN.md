# 使用示例

## 适用范围

这些示例是为了强化本仓库“分阶段、边界清晰、小步推进”的使用方式。

它们不是“一句话生成整个产品设计”的提示词。

## 示例 1：先做结构，不急着精修

适用场景：
页面还很新，或者需求仍然模糊。

提示词：

> 先规划页面结构，再只搭 skeleton。先不要做视觉精修。范围限制在一个 landing page 内。

好的输出应体现：
- 明确 planning 阶段
- section 顺序清楚
- 只搭结构，不抢跑 refinement

## 示例 2：只改一个 section

适用场景：
整页结构应该保持稳定，只改局部。

提示词：

> 只重做 Hero 区，不要动页面其他部分。提出修改前先重新读取当前结构。

好的输出应体现：
- 只处理一个 section
- 改前先读结构
- 明确边界
- 不牵连其他区块

## 示例 3：先审查，再决定改不改

适用场景：
当前设计可能已经接近可用，但质量不确定。

提示词：

> 先审查 spacing、字体层级和 token 一致性，再决定是否编辑。如果结构稳定，只提出 refinement。

好的输出应体现：
- audit-first
- 先列问题再改
- 明确区分 refinement 与重构

## 示例 4：MCP 不可用时的降级处理

适用场景：
MCP 可能不可用，但任务仍然可以继续推进。

提示词：

> MCP 可能不可用。如果 live editing 没准备好，就切换到 fallback 模式，输出按 section 划分的执行计划，不要假装自己已经实时修改。

好的输出应体现：
- 明确声明 fallback
- 不假装 live edit 成功
- 仍然给出可执行下一步

## 示例 5：受控 refinement

适用场景：
结构已经确认，只做视觉细节和一致性提升。

提示词：

> skeleton 已经确认。现在只做 spacing、对齐、强调层级和一致性 refinement，不要改信息架构。

好的输出应体现：
- 只做 refinement
- 保持结构不变
- 聚焦视觉一致性
